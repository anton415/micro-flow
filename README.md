## Цель
Разработать демонстрационный проект микросервисной архитектуры, развернутый в Kubernetes, который иллюстрирует базовые принципы взаимодействия сервисов через сеть и ingress.

## Функциональные требования

### 1. Инфраструктура

* Развернуть локальный Kubernetes-кластер (например, Minikube или Kind).
* Все компоненты системы должны работать внутри кластера.
* Использовать Kubernetes как основную платформу оркестрации.

### 2. Микросервисы

#### 2.1 Сервис заявок (Request Service)

Функции:

* Принимает HTTP-запросы (например, POST /requests).
* Регистрирует заявку (в памяти или простой БД — можно in-memory).
* После регистрации отправляет событие/запрос в сервис нотификаций.

Требования:

* REST API
* Логирование входящих запросов
* Генерация ID заявки

#### 2.2 Сервис нотификаций (Notification Service)

Функции:

* Принимает запрос от сервиса заявок.
* Сохраняет факт получения нотификации (например, в память или лог).
* Может просто писать в лог (этого достаточно для демо).

Требования:

* Внутренний API (не обязательно публичный)
* Логирование событий

### 3. Взаимодействие сервисов

* Сервис заявок должен обращаться к сервису нотификаций:
    * через DNS внутри Kubernetes (например: http://notification-service)
* Без прямого IP — только через сервисы Kubernetes

### 4. Ingress

* Настроить Ingress Controller
* Обеспечить внешний доступ к сервису заявок:
    * например: http://localhost/requests
* Входящий трафик должен идти через Ingress → Request Service

### 5. Минимальный UI (обязательное требование)

Цель UI

Не “красота”, а проверяемость системы.

Требования:

* Статический фронтенд (HTML + JS)
* Размещается через GitHub Pages
* Делает:
    * форму отправки заявки (POST)
    * кнопку “создать заявку”
    * вывод ответа от сервера

Архитектура:

GitHub Pages → обращается к Ingress endpoint (т.е. к твоему backend)

Важно:

* CORS должен быть настроен на backend
* UI не должен зависеть от backend-деплоя (разные хосты)


### 6. Нефункциональные требования

* Контейнеризация:
    * Каждый сервис должен быть упакован в Docker
* Kubernetes манифесты:
    * Deployment
    * Service
    * Ingress
* Логирование:
    * Через stdout (kubectl logs)
* Простота:
    * без Kafka / RabbitMQ (если не требуется)
    * синхронный HTTP — достаточно


### 7. Критерии готовности (Definition of Done)

Проект считается завершённым, если:

1. Кластер поднимается локально
2. Оба сервиса задеплоены в Kubernetes
3. Ingress работает и доступен извне
4. UI (GitHub Pages):
    * отправляет запрос
    * получает ответ
5. При создании заявки:
    * Request Service обрабатывает её
    * Notification Service получает событие
6. Это подтверждается логами

### 8. Архитектурная схема

    [ GitHub Pages UI ]
          ↓
      [ Ingress ]
          ↓
    [ Request Service ]
          ↓ (HTTP через DNS)
    [ Notification Service ]
